1. Planning Evidence

1.1 Project Brief & Role

The brief required a professional portfolio website of at least five pages, built using HTML, CSS and JavaScript. My portfolio targets UX design roles — specifically product and interaction design positions in digital and tech companies. The primary audiences are hiring managers, creative directors, and potential freelance clients.

I chose a dark-themed, single-brand design system to create a strong visual identity, and decided that all pages should share a unified navigation, footer, and external stylesheet — a deliberate architecture decision made before writing a single line of code.

1.2 Site Map

Before building I mapped out the full structure: what pages exist, what each contains, and how navigation connects them. The site is flat (no sub-pages) to keep navigation simple and all content reachable in one click.

1.4 Design Decisions

Three key decisions were locked before coding:

  • Layout: CSS Grid for page-level two-column layouts; Flexbox for navigation, cards, and inline elements. No layout frameworks — entirely hand-written.
  • Typography: Space Grotesk (geometric, strong personality) for headings; Inter (neutral, highly legible) for body text. Both loaded via Google Fonts for performance (preconnect headers added).
  • Colour: Deep dark background (#07070f) with a pink accent (#ec4899) for energy and contrast. Light mode toggled via data-theme attribute on <html>.

Stage 02 10 April 2026 — 14:30

CSS design system — custom properties & reset

Rather than hardcoding colour values throughout the stylesheet, I built a design token system using CSS custom properties on :root. This was a deliberate architectural choice: it allowed the entire colour scheme to be changed later (which it was — from purple to pink) by editing only a handful of lines.

Code excerpt — CSS custom properties (design tokens)

:root {
  /* Colours */
  --c-bg:           #07070f;
  --c-surface:      #0e0e1c;
  --c-accent:       #ec4899;       /* pink accent */
  --c-accent-light: #f9a8d4;
  --c-accent-glow:  rgba(236, 72, 153, 0.22);
  --c-text:         #f0f0f6;
  --c-text-2:       #8080a0;
  --c-border:       rgba(255, 255, 255, 0.07);

  /* Typography */
  --font-heading: 'Space Grotesk', sans-serif;
  --font-body:    'Inter', sans-serif;

  /* Motion */
  --ease: cubic-bezier(0.4, 0, 0.2, 1);
  --dur:  0.3s;
}

/* Light theme overrides — toggled via JS on <html data-theme> */
[data-theme="light"] {
  --c-bg:      #f5f5fb;
  --c-surface: #ffffff;
  --c-text:    #0d0d1c;
  --c-text-2:  #555572;
}

Why this mattered

The [data-theme="light"] override block meant that the entire colour scheme could flip with a single attribute change on the root element — no inline styles, no JavaScript class manipulation on individual components. This is the modern, accessible approach to theming.

Stage 03 14 April 2026 — 11:00

Navigation — sticky header, mobile hamburger menu, active link

The navigation was the most technically involved single component — it needed to be fixed to the top, apply a frosted-glass style after scrolling, collapse to a hamburger menu on mobile, and highlight the current page link.

Code excerpt — sticky nav JavaScript (from main.js)

/* Sticky nav — adds frosted-glass class when user scrolls */
(function initNav() {
  var nav = document.querySelector('.nav');
  if (!nav) return;

  function onScroll() {
    nav.classList.toggle('nav--solid', window.scrollY > 16);
  }

  window.addEventListener('scroll', onScroll, { passive: true });
  onScroll(); // run on load to handle page refresh mid-scroll
}());

Code excerpt — active nav link (marks current page)

/* Reads the filename from the URL and marks the matching nav link */
(function markActive() {
  var page = window.location.pathname.split('/').pop() || 'index.html';
  document.querySelectorAll('.nav__link').forEach(function (a) {
    if (a.getAttribute('href') === page) {
      a.setAttribute('aria-current', 'page');
    }
  });
}());

Code excerpt — CSS frosted glass effect on scroll

.nav--solid {
  background: rgba(7, 7, 15, 0.82);
  backdrop-filter: blur(18px);
  -webkit-backdrop-filter: blur(18px);
  border-bottom-color: var(--c-border);
}

Why this mattered

Using aria-current="page" instead of a custom CSS class is the semantically correct, accessible approach — screen readers announce the current page to users, and the CSS underline indicator is driven by the same attribute.

Typing / typewriter effect (JavaScript)

The animated tagline cycles through phrases using a custom typewriter function. The interaction pattern was inspired by the hero section of Linear.app (https://linear.app), which uses cycling animated text to communicate multiple product benefits without cluttering the headline. The implementation below is entirely original code written for this project. 

/* Typing effect — cycles through phrases with typewriter animation */
(function initTyping() {
  var el = document.querySelector('[data-typing]');
  if (!el) return;

  var phrases = [
    'experiences people love.',
    'clarity from complexity.',
    'seamless digital journeys.',
    'interfaces that just work.'
  ];

  var pi = 0, ci = 0, deleting = false;

  function tick() {
    var phrase = phrases[pi];
    el.textContent = deleting ? phrase.slice(0, ci - 1) : phrase.slice(0, ci + 1);
    deleting ? ci-- : ci++;

    var wait = deleting ? 38 : 76;

    if (!deleting && ci === phrase.length) {
      wait = 2200;       // pause at end of phrase
      deleting = true;
    } else if (deleting && ci === 0) {
      deleting = false;
      pi = (pi + 1) % phrases.length;  // move to next phrase
      wait = 420;
    }
    setTimeout(tick, wait);
  }

  tick();
}());

Scroll reveal — Intersection Observer

Elements with the class .reveal animate in when they enter the viewport, using the modern IntersectionObserver API rather than a scroll event listener. This is more performant because it runs off the main thread. 

(function initReveal() {
  var els = document.querySelectorAll('.reveal');
  if (!els.length) return;

  var io = new IntersectionObserver(function (entries) {
    entries.forEach(function (e) {
      if (e.isIntersecting) {
        e.target.classList.add('in');  // triggers CSS transition
        io.unobserve(e.target);        // animate once only
      }
    });
  }, { threshold: 0.12, rootMargin: '0px 0px -40px 0px' });

  els.forEach(function (el) { io.observe(el); });
}());

Reference for the Intersection Observer API: https://developer.mozilla.org/en-US/docs/Web/API/Intersection_Observer_API

Code excerpt — modal accessibility

/* Focus trap: save element that opened modal, return focus on close */
var focusBefore;

function open(card) {
  /* ... populate content ... */
  modal.classList.add('open');
  document.body.style.overflow = 'hidden';
  focusBefore = document.activeElement;         // remember focus origin
  modal.querySelector('.modal__close').focus(); // move focus into modal
}

function close() {
  modal.classList.remove('open');
  document.body.style.overflow = '';
  if (focusBefore) focusBefore.focus();         // return focus
}

/* Close on Escape key */
document.addEventListener('keydown', function (e) {
  if (e.key === 'Escape' && modal.classList.contains('open')) close();
});

Why this mattered

Returning keyboard focus to the triggering element when the modal closes is a WCAG 2.1 requirement (Success Criterion 2.4.3). Without it, keyboard and screen reader users lose their place in the page.

Stage 06 20 April 2026 — 09:00

Contact form — validation, success state & accessibility

The contact form includes full client-side validation written in vanilla JavaScript: name length, email pattern matching, and message length checks. Errors are announced to screen readers via role="alert" and linked to inputs via aria-describedby.

Code excerpt — form validation

form.addEventListener('submit', function (e) {
  e.preventDefault();
  clearErrors();
  var ok = true;

  /* Email validation using regex */
  var email = form.querySelector('#f-email');
  if (!email || !/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email.value.trim())) {
    markErr(email, 'Enter a valid email address.');
    ok = false;
  }

  if (ok) {
    form.style.display = 'none';
    document.querySelector('.form-success').classList.add('show');
  }
});

function markErr(input, text) {
  input.classList.add('err');
  /* data-err attribute links the error span to its input by ID */
  var el = document.querySelector('[data-err="' + input.id + '"]');
  if (el) { el.textContent = text; el.classList.add('show'); }
}

Code excerpt — accessible error markup (HTML)

<input
  class="form-input"
  type="email"
  id="f-email"
  aria-required="true"
  aria-describedby="err-email"
/>
<span class="form-err" id="err-email"
      data-err="f-email"
      role="alert"></span>

Why this mattered

Connecting errors to their inputs via aria-describedby means screen readers read the error message immediately after the field label when the user navigates to an invalid input — exactly the behaviour WCAG 2.1 Success Criterion 3.3.1 (Error Identification) requires.

3. Reflective Discussion

Problems faced and how they were solved

The most technically challenging problem was building an accessible modal lightbox from scratch. My first implementation opened and closed the modal correctly, but keyboard users could tab behind it while it was open — meaning the modal overlay was purely visual and not truly accessible. After researching ARIA modal patterns, I added focus management: saving the element that triggered the modal, moving focus to the close button when the modal opens, and returning focus to the original element on close. I also added an Escape key listener and a backdrop click handler, both calling the same close() function to avoid code duplication. This solved the accessibility gap entirely.

A second problem was the sticky contact info column on the contact page. On tablet screens, the two-column grid collapsed to a single column but the position: sticky property remained active — causing the info panel to overlap content below it. The fix was to set position: static inside the @media (max-width: 1024px) breakpoint, which I had already defined for the layout collapse. A small fix, but one that would have been invisible in desktop testing and broken on every tablet visitor.

A decision I am pleased with

I am most pleased with the CSS custom property (design token) architecture. By defining all colours, fonts, spacing, and easing values as variables on :root, I could change the entire colour scheme from violet to pink by updating fewer than ten values in a single file — a change that would otherwise require a find-and-replace across the entire codebase. It also made the light/dark theme toggle trivial: a second block overrides only the colour tokens, and JavaScript simply toggles a data-theme attribute on the <html> element. This is the approach used by major design systems and I am glad I adopted it from the start.

What I would improve with more time

With more time, I would add genuine case study pages — individual HTML pages for each portfolio project with full research documentation, annotated wireframes, and before/after comparisons. The current modal approach works well as a placeholder but limits the depth of storytelling available for each project.

I would also add a proper skip-to-content link at the very top of each page (a standard accessibility pattern for keyboard users who want to bypass navigation and jump straight to the main content), and implement prefers-reduced-motion media queries to disable or slow all animations for users who have configured their operating system to reduce motion.
